浅谈操作系统虚拟内存

Posted on 2016-12-15 | In Operating System , 内存管理

  上篇讲到OS内存管理，方式是将整个program load到memory里，这么做浪费了memory空间，因为一个program里面有些程序是很少用到的，有些数组开了很大用的很少，所以，高效率的做法是将需要的page放到memory里面，平时我们玩大型游戏，十几G的游戏能在4G的内存运行良好，就是这么一个道理，虚拟内存，简单说，就是利用disk，当内存不够用时候，就把一部分进程踢到disk里面去，需要的时候，再从disk拿回来。
  page table记录着一个page是否在physical memory里面，如果不在，就会发生page fault，OS就会从disk里面将需要的page load到physical memory里面，这个过程耗费时间，page fault rate越高，时间浪费的就越多，EAT（average access time）就越大。
  假如我们采用了某种算法，选择了一个要被踢到disk的进程A，前脚踢出A，后脚又要用到A，于是又swap in进来，那么这么做是很低效的，频繁地swap in，swap out消耗了大量的时间。一个理想的情况是，我们把将来最不会用到进程swap out出去。可是，没人能够预测将来，所以，我们只能根据过去来预测将来，判断应该把谁swap out出去。

OS内存管理

Posted on 2016-12-15 | In Operating System , 内存管理

浅谈操作系统内存管理

连续内存分配

我们每天都在run很多的进程，这些进程在内存中处在什么位置呢？我们该怎么样根据这些进程的逻辑地址对应到实际地址呢？这是操作系统要面对的一个问题。
直观的，我们会考虑把一个进程放入一段连续的内存中，一个进程就是一个整体，进程内部无内存空洞，进程和进程之间可能存在内存空洞，当进程run完后，它会释放占用的内存，这些被释放的内存也会形成了空洞
空洞的存在是对资源的浪费，为了充分利用这些内存空洞，我们考虑，将后续的进程填入这些空洞中，有三种填补方法：first fit, best fit, worse fit，first fit就是找到第一个能够放入的空洞，放入进程，best fit就是遍历所有的空洞，找到最合适的，所谓最合适，就是恰到好处，空洞长度比进程的长度长最小，worse fit与此相反，first fit速度最快，best fit内存利用率最高。
但是，尽管如此，内存还是会有很多的小洞（fragmentation）没办法被填满，平均情况下，会有三分一的内存被浪费了！怎么办？
